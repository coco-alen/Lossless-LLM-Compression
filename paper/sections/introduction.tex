\section{Introduction}
\label{sec:intro}

Large language model (LLM) serving increasingly separates the prefill phase, which processes the input prompt in parallel, from the decode phase, which generates tokens autoregressively~\cite{distserve,splitwise,mooncake}. This disaggregated prefill-decode (PD) design improves hardware utilization because prefill and decode have different compute and memory behavior. The separation, however, turns the KV cache into an explicit cross-node payload: the prefill worker must move all keys and values to the decode worker before generation can continue.

The payload is large enough to dominate time-to-first-token (TTFT). For a Llama-3-70B-style model at 64K context, the BF16 KV cache is roughly 21.5~GB across 80 layers. Even with high-end cross-node links, moving that cache can add hundreds of milliseconds before the first decoded token appears. Existing PD serving systems therefore optimize routing, scheduling, memory placement, and transfer overlap, but they commonly still move KV tensors in their native BF16 or FP8 representation.

Compression appears attractive, but straightforward alternatives do not match the serving path:

\begin{itemize}[nosep,leftmargin=*]
\item \textbf{Generic lossless codecs} see BF16 tensors as high-entropy byte streams. In our measurements, LZ4 gives essentially no reduction on BF16 KV caches, while zstd level~1 is CPU-bound and too slow for the critical path.
\item \textbf{Lossy KV quantization} can halve the payload with FP8, but it changes the tensor values. That is useful in many deployments, yet it complicates exact replay, debugging, and correctness-sensitive serving.
\item \textbf{Weight-oriented lossless codecs} exploit floating-point structure, but they are designed for model storage and loading rather than layer-pipelined KV transfer between PD workers.
\end{itemize}

We observe that KV caches contain a simple format-level regularity that these alternatives miss. In BF16, the 8-bit exponent field of KV tensors concentrates on a small set of values: the top 15 exponents cover 99.1--99.8\% of elements across the models we profile, including both dense and MoE LLMs. The sign bit and mantissa remain high-entropy, so \splitzip leaves them uncompressed and only encodes the exponent stream.

This paper presents \splitzip, a format-aware KV-cache codec designed for disaggregated serving. The primary BF16 mode maps frequent exponents to 4-bit codes, packs two codes per byte, and stores rare exponents in an escape stream. Since the sign and mantissa bytes are kept verbatim and escaped exponents are restored exactly, decoding reconstructs the original BF16 KV cache bit for bit. The same design can also be applied to native FP8 KV caches, where it gives additional lossless compression relative to the FP8 representation.

Our main contributions are:

\begin{enumerate}[nosep,leftmargin=*]
\item \textbf{KV exponent characterization.} We profile BF16 KV caches across six model settings, including Qwen, Llama, Phi, and Qwen3-MoE, and show that exponent concentration is stable enough for fixed-width coding.

\item \textbf{A lossless GPU codec for BF16 KV transfer.} \splitzip combines 4-bit exponent coding with an escape stream, preserving exact BF16 values while keeping the GPU kernels simple and bandwidth-oriented.

\item \textbf{Native BF16 and FP8 evaluation.} We evaluate raw BF16, BF16+\splitzip, native FP8 E4M3/E5M2, and FP8+\splitzip to separate exact compression benefits from lossy precision reduction.

\item \textbf{Serving-path validation.} We integrate \splitzip with Mooncake Transfer Engine and measure speedups on both controlled layer-size transfers and real Qwen3-30B-A3B KV caches, while validating exact reconstruction and generation fidelity.
\end{enumerate}
