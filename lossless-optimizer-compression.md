# Lossless optimizer state compression remains an open frontier

**Truly lossless compression of Adam's m1/m2 buffers in live GPU memory during training does not yet exist.** Every practical method that reduces optimizer state memory — 8-bit Adam, GaLore, APOLLO, COAT — is technically lossy, though several achieve "functionally lossless" results where no measurable quality degradation occurs. Meanwhile, the entropy-coding approach that DFloat11 uses to losslessly compress BF16 weights (exploiting low exponent entropy) has strong theoretical justification for working on optimizer states too — COAT and ZipNN both confirm that Adam's moment buffers have narrow dynamic ranges and concentrated exponent distributions — but no one has yet built a GPU-efficient implementation that applies this to live optimizer states. This represents a genuine gap in the literature: the algorithmic pieces exist, but the systems engineering to make it practical on GPUs during training remains undone.

## The memory problem: optimizer states dwarf the model itself

For a model with Ψ parameters trained with AdamW in mixed precision, the optimizer demands **12Ψ bytes** — 4 bytes each for FP32 copies of the first moment (m), second moment (v), and master weights. The BF16 working copy adds only 2Ψ bytes. Optimizer states thus consume **6× the working model's memory**. For a 7B-parameter LLM, that's roughly **84 GB** of optimizer state alone, making it the dominant memory bottleneck.

This overhead has spawned two fundamentally different classes of solutions. **Distribution approaches** (DeepSpeed ZeRO, PyTorch FSDP) shard states across GPUs without compression — each GPU holds 1/N of the full-precision states. **Compression approaches** reduce the per-element storage cost but invariably sacrifice bit-exact fidelity. A third category — offloading to CPU/NVMe — moves states to cheaper memory tiers but introduces PCIe bandwidth bottlenecks. None of these constitutes true compression in the information-theoretic sense.

## The lossy compression landscape: from 8-bit to SGD-level memory

The practical toolkit for reducing optimizer memory has grown rapidly since 2021, with methods spanning a wide spectrum of aggressiveness and fidelity:

**Quantization-based methods** directly reduce the bit-width of stored states. Tim Dettmers' **8-bit Adam** (ICLR 2022, arXiv 2110.02861) pioneered this with block-wise dynamic quantization, achieving **4× compression** of m and v with empirically zero degradation across models up to 1.5B parameters. The **4-bit optimizer** work by Li, Chen, and Zhu (NeurIPS 2024, arXiv 2309.01507) pushed further with rank-1 normalization and a zero-point-free quantizer for v, reaching **8× compression** with marginal accuracy loss. **COAT** (ICLR 2025, arXiv 2410.19313) quantizes both moments to FP8 using dynamic range expansion, delivering **1.54× total memory reduction** with near-lossless LLaMA pretraining. Most recently, **FlashOptim** (2025, arXiv 2602.23349) from Databricks applies companding functions — nonlinear preprocessing that reshapes heavy-tailed optimizer distributions before linear quantization — to reduce AdamW from 16 to **7 bytes per parameter** with "no measurable quality degradation."

**Low-rank and structural methods** reformulate the optimizer algorithm itself. **Adafactor** (ICML 2018) factorizes v into row and column vectors, cutting second-moment memory from O(mn) to O(m+n). **GaLore** (ICML 2024, arXiv 2403.03507) projects gradients into a low-rank subspace via SVD, maintaining m and v only in that subspace for up to **65.5% optimizer memory reduction** — the first method to pretrain LLaMA-7B on a single RTX 4090. **Adam-mini** (arXiv 2406.16793) exploits the block-diagonal Hessian structure of transformers to share a single learning rate per parameter block, cutting v storage by **~50%**. **APOLLO** (MLSys 2025, arXiv 2412.05270) represents the current frontier: using pure random projection (no SVD) for structured learning-rate scaling, APOLLO-Mini achieves **SGD-level memory** while matching AdamW performance across models from 60M to 13B parameters.

A critical distinction among lossy methods is whether they provide **convergence guarantees**. **MicroAdam** (NeurIPS 2024, arXiv 2405.15593) uses TopK gradient sparsification with compressed error feedback and proves convergence at AMSGrad's rate. **LDAdam** (ICLR 2025, arXiv 2410.16103) combines low-rank projection with a generalized error feedback mechanism, providing the first formal convergence proof for methods that compress both gradients and optimizer states. These error-feedback approaches are the closest to "reversible" compression — they provably recover the full optimization trajectory asymptotically.

## DFloat11 and the case for entropy coding optimizer states

DFloat11 (NeurIPS 2025, arXiv 2504.11651), developed at Rice University, achieves **~30% lossless compression** of BF16 model weights by Huffman-coding the exponent field, which carries only **~2.6 bits of entropy** out of 8 bits of capacity. A companion theoretical paper (ECF8, arXiv 2510.02676) explains this exponent concentration as arising from the heavy-tailed α-stable distributions that SGD dynamics induce in neural network weights. **NeuZip** (arXiv 2410.20650) applies the same insight using Asymmetric Numeral Systems (ANS) instead of Huffman coding, reducing LLaMA-3 8B training memory from 31 GB to under 16 GB — with a truly lossless variant that produces **bit-for-bit identical training dynamics**.

The natural question is whether this entropy-coding approach transfers to optimizer states, and multiple lines of evidence suggest it should. COAT's analysis reveals that Adam's first moment has a dynamic range of only **~10³** and the second moment only **~10¹**, compared to FP32's total range of ~10⁷⁷ — meaning the exponent field of optimizer states is even more concentrated than that of weights. ZipNN (arXiv 2411.05239) explicitly analyzed optimizer state compressibility during RoBERTa training and confirmed that exponent bytes have the highest compressibility, with steadily increasing redundancy as training progresses. The **Single-Stage Huffman Encoder** paper (arXiv 2601.10673) proposes pre-computed Huffman codebooks applicable to all ML tensor types including optimizer states, achieving within 0.5% of per-shard optimal coding.

Yet **no published work applies DFloat11-style entropy coding to Adam's m1/m2 buffers in live GPU memory**. The reason is a systems-level obstacle, not an algorithmic one: variable-length entropy codes break the O(1) random-access pattern that GPU optimizer kernels require. DFloat11 solved this for inference by decompressing transformer blocks just before matrix multiplications, but the optimizer step accesses every element of m and v during a single fused kernel. Designing a GPU-efficient decompression scheme compatible with element-wise optimizer updates is the key unsolved engineering challenge.

## Checkpoint compression: lossless exists, but only for storage

For offline storage and transfer (not live GPU memory), lossless compression of optimizer states is a solved problem. **LMC** (arXiv 2505.09810) achieves true lossless compression of full checkpoint tensors — including optimizer states — using byte-grouping, Huffman encoding, and incremental delta compression between snapshots, with throughput of **2.78 GiB/s compression and 3.76 GiB/s decompression**. ZipNN achieves ~33% lossless compression on BF16 model files. **ExCP** (ICML 2024, arXiv 2406.11257) exploits weight-momentum joint structure for near-lossless checkpoint compression at 25–70×, though it uses quantization and pruning.

Delta coding between consecutive checkpoints is particularly effective because Adam's exponential moving averages change slowly — most bytes remain identical between steps. **BitSnap** (arXiv 2511.12376) found that only ~15% of parameter bytes change between adjacent checkpoints, enabling **5× lossless compression** through bitmask-based delta encoding. However, the authors explicitly rejected Huffman encoding for raw delta data because its entropy was too high without preprocessing — suggesting that naive entropy coding of optimizer state *deltas* is insufficient without the byte-grouping/exponent-extraction preprocessing that DFloat11 and ZipNN use.

## The landscape: what's solved, what's open, and what the tradeoffs are

The field has effectively converged on a practical consensus: **lossy compression at 8-bit or above is "good enough" for most training scenarios**, with methods like 8-bit Adam, COAT, and FlashOptim achieving no measurable quality loss. Below 8 bits, methods like 4-bit optimizers and APOLLO-Mini offer dramatically more savings but with increasing risk of subtle convergence effects, particularly for pretraining from scratch versus fine-tuning.

Three problems remain genuinely open:

- **Lossless in-memory compression.** No method losslessly compresses FP32 optimizer states in GPU memory during training. The entropy-coding approach is theoretically sound — optimizer states have low exponent entropy and narrow dynamic ranges — but the GPU systems engineering (random-access decompression in fused optimizer kernels) is unsolved. This is likely achievable with custom CUDA kernels following DFloat11's approach, but would require non-trivial engineering to handle the optimizer step's access patterns.

- **Information-theoretic characterization.** No published work measures the actual entropy of Adam's m1/m2 buffers across training, which would establish the theoretical compression ceiling. COAT's dynamic-range analysis and ZipNN's compressibility plots provide indirect evidence, but a rigorous entropy analysis — analogous to ECF8's theoretical treatment of weight distributions — is missing.

- **Temporal compression during training.** Adam's moments are exponential moving averages with configurable decay (β₁=0.9, β₂=0.999), meaning consecutive-step values are highly correlated. Delta encoding between optimizer steps — compressing only the change to m and v at each step — could achieve substantial lossless compression, especially for v which changes extremely slowly. No work exploits this temporal redundancy for live memory reduction.

The fundamental tradeoff is **memory savings versus engineering complexity and runtime overhead**. Lossy methods are simple, fast, and battle-tested. Lossless entropy coding would preserve exact training dynamics (critical for reproducibility, debugging, and checkpoint resumability), but adds decompression latency to a memory-bandwidth-bound kernel. For most practitioners, 8-bit Adam's "functionally lossless" behavior at 4× compression is the pragmatic sweet spot. But for those who need bit-exact reproducibility or who want to push compression further without any convergence risk, lossless optimizer compression remains an open and promising research direction — one where the theoretical foundations are already in place, waiting for the right GPU implementation.
